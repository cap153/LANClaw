### 多用户 Session 管理

```
pi --session-dir ~/.local/share/lanclaw/sessions/
```

每个用户的 session 文件由 pi 管理在指定目录中。LANClaw 通过 RPC 命令切换：

┌───────────────┬────────────────────────────────────────────────────────────────────────────────────────────────┐
│ 场景          │ RPC 命令序列                                                                                   │
├───────────────┼────────────────────────────────────────────────────────────────────────────────────────────────┤
│ 用户 A 发消息 │ switch_session <A的session> → prompt "..." → 等 agent_end → get_last_assistant_text → 提取回复 │
├───────────────┼────────────────────────────────────────────────────────────────────────────────────────────────┤
│ 用户 B 发消息 │ switch_session <B的session> → prompt "..." → ...                                               │
├───────────────┼────────────────────────────────────────────────────────────────────────────────────────────────┤
│ /new          │ switch_session <A的session> → new_session                                                      │
├───────────────┼────────────────────────────────────────────────────────────────────────────────────────────────┤
│ 首次用户      │ 直接 prompt（pi 自动创建 session 文件）                                                        │
└───────────────┴────────────────────────────────────────────────────────────────────────────────────────────────┘

### 事件提取回复的两种方案

方案 A（推荐，简单可靠）：
1. 发 prompt 命令
2. 等待 agent_end 事件（期间可忽略或计数 text_delta 做进度）
3. 发 get_last_assistant_text 获取最终文本
4. 解析 response 的 data.text

方案 B（流式，更实时）：
1. 发 prompt 命令
2. 监听 message_update 的 text_delta，累积到 buffer
3. 监听 agent_end，buffer 就是完整回复

采用方案 A，多一次 get_last_assistant_text 但实现简单健壮。

### 并发控制

RPC 进程是单线程处理消息的。当多用户同时发送消息时：

1. 用一个 tokio::sync::Mutex 串行化所有 RPC 操作
2. 内部维护一个队列：当前正在处理用户 A 时，用户 B 的消息排队等待
3. 队列长度警告（防止堆积过多）

```rust
pub async fn prompt(&self, user_id: &str, message: &str, files: &[PathBuf]) -> Result<String> {
   let _lock = self.mutex.lock().await;

   // 1. 切换到用户的 session
   self.send_cmd(&switch_session_cmd(user_id)).await;
   self.wait_response().await;

   // 2. 发送 prompt
   self.send_cmd(&prompt_cmd(message)).await;
   self.wait_response().await;

   // 3. 等待 agent_end
   self.wait_event("agent_end").await;

   // 4. 获取最终文本
   self.send_cmd(&get_last_assistant_text()).await;
   let resp = self.wait_response().await;

   Ok(resp.data.text)
}
```

### RPC 子进程生命周期

```
LANClaw 启动
 ├─ spawn pi --mode rpc --session-dir <dir> --no-context-files --skill <path>
 ├─ 等待 RPC 进程就绪（发送 get_state 确认响应正常）
 ├─ 进入主循环
 │
 ├─ 正常退出 → 发送 abort（如有进行中），关闭 stdin，wait
 ├─ RPC 崩溃 → 自动重启（重启后重新处理当前队列）
 └─ RPC 卡死 → 超时 kill 后重启
```

### Pi 命令行变化

```
改造前:
 pi -p --session <user>.jsonl --model <m> --skill <s> --no-context-files "你好"

改造后（仅启动时一次）:
 pi --mode rpc --session-dir ~/.../sessions --model <m> --skill <s> --no-context-files
```

不再传 --thinking 给 pi 启动参数，thinking 改为在 prompt 阶段决定是否传（但目前 pi RPC 不支持 per-prompt thinking，所以启动时传比较好）。

### 进度显示优化

由于 RPC 是流式的，可以在等待回复时给用户一个"正在输入"的反馈：

1. 收到 tool_execution_start 事件 → 可发送 "正在思考…"
2. 收到 message_update / text_delta → 可发送流式文本
3. 收到 agent_end → 发送完整回复
